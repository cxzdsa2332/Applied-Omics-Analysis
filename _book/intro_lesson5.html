7  探索生物大数据的基石：数据库网络与核心文件格式

Published

April 10, 2026

7.1 引言：数据的汪洋与打水的桶

在开始敲代码之前，我们必须回答两个问题：“全世界的生物数据存放在哪里？” 以及 “我们把数据下载到本地后，它们是以什么面貌存在的？” 生物信息学是一个高度强调“开源与共享”的学科。任何一篇顶级期刊论文发表的前提，都是必须将原始测序数据上传到全球公共数据库中。今天，我们将先认识这些全球数据枢纽，再深入拆解承载这些海量信息的核心文件格式。

7.2 宏观世界：全球三大生物数据基础设施

在计算科学中，我们可以把这些数据库理解为生物学界的 AWS S3 集群，它们共同构成了 INSDC（国际核酸序列数据库合作组织，International Nucleotide Sequence Database Collaboration）。INSDC 的三个协作伙伴（NCBI、EBI、DDBJ）采用24 小时同步机制：任何一个数据库收到的新数据，必须在 24 小时内同步到其他两个伙伴。这确保了全球研究者无论向哪个数据库提交，都能被全世界同步感知。

7.2.1 NCBI（美国国家生物技术信息中心）

这是生物信息学领域的绝对霸主，相当于生物学界的 Google。它旗下有几个我们需要极其熟悉的子数据库： * SRA (Sequence Read Archive)：原始数据的垃圾场与金矿。这里存放着全球所有的原始高通量测序数据（FASTQ 格式的压缩版）。数据量极其庞大（数十 PB 级别）。 * GEO (Gene Expression Omnibus)：处理后的矩阵仓库。研究者会将他们算好的“基因表达量矩阵”（通常是常规的 Excel/CSV 表格或 HDF5 文件）传到这里。这是挖掘现成数据最常去的地方。 * GenBank / RefSeq：标准参考答案。存放着经过人类专家严格审核的、高质量的物种参考基因组序列。

7.2.2 EBI（欧洲生物信息学研究所）与 Ensembl

Ensembl 是 EBI 旗下的王牌基因组浏览器项目。如果说 NCBI 偏向于“收纳一切”，Ensembl 则在基因组注释（Annotations，即定义基因的坐标位置）方面做得极其精准和结构化，是下载参考基因组 GTF 注释文件的首选之地。

7.3 文本的哲学：序列与坐标格式（ASCII）

从数据库下载数据后，我们最先接触到的是纯文本（Plain Text）格式。它们牺牲了存储效率，但换取了极高的“人类可读性”。

7.3.1 FASTA（.fasta / .fa）：参考基因组的字典

最古老、最基础的序列存储格式，本质是一个键值对（Key-Value Pair）集合。 * Key：以 > 开头，包含序列 ID（如染色体编号）和可选的描述信息。 * Value：紧随其后的纯碱基（A/T/C/G）或氨基酸序列。

>chr1 Homo sapiens chromosome 1, GRCh38
NNNNNNNNNNNNNNNTAACCCTAACCCTAACCCTAACCCTAACCCTAACCCTA...
>chr2 Homo sapiens chromosome 2, GRCh38
NNNNNNNNNNNNNNNTAACCCTAACCCTAACCCTAACCCTAACCCTAACCCTA...

关键特性： - 极其简洁，易于解析 - 每行通常限制在 50-80 字符以提高可读性 - 广泛用于参考基因组、蛋白序列库（如 UniProt）

7.3.2 FASTQ（.fastq / .fq）：原始测序数据的真实面貌

当高通量测序仪器（Illumina、PacBio 等）输出数据时，每条读取片段（read）都包含序列本身和质量评分。FASTQ 格式正是为此而生。

@SRR1234567.1 reads from sample A
AGCTTAGCTAGCTACGTACGTACGTACGTACGTACGTACGTACGTACGTACGTAC
+
IIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIII
@SRR1234567.2 reads from sample A
GCTAGCTAGCTAGCTAGCTAGCTAGCTAGCTAGCTAGCTAGCTAGCTAGCTAGC
+
HHHHHHHHHHHHHHHHGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGFFFFFCCCC

四行结构： 1. 行 1：@ 前缀 + 读取 ID + 可选描述 2. 行 2：实际序列 3. 行 3：+ 分隔符（有时后跟 ID，有时为空） 4. 行 4：质量分数（与序列等长）。采用 Phred 质量评分系统：ASCII 字符编码表示概率

质量分数的秘密： - Phred 质量 Q = -10 × log₁₀(P)，其中 P 是碱基错误的概率 - 字符 I (ASCII 73) ≈ Q30，表示 0.1% 错误率（极高质量） - 字符 # (ASCII 35) ≈ Q2，表示 63% 错误率（极低质量） - 一般接纳的阈值：Q20（1% 错误率）或 Q30

这就是为什么原始数据文件的大小达到 数十到数百 GB——每个碱基都附带质量信息。

延伸阅读：Phred 质量分数系统的详细原理和 Illumina 测序流程，已在 第二节·高通量测序原理 中详细介绍。

7.3.3 SAM / BAM / CRAM：序列比对的金字塔

从 FASTQ 文件出发，下一步是将这些短读取与参考基因组进行比对（alignment）。这个过程输出 SAM（Sequence Alignment Map）格式。

SAM 格式（可读文本版）：

@HD VN:1.0  SO:coordinate
@SQ SN:chr1 LN:248956422
read1   0   chr1    1000    60  50M *   0   0   AGCTTAGCTAGCTACGTACGTACGTACGTACGTACGTACGTACGTACG    IIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIII
read2   16  chr1    2000    30  40M10S  *   0   0   GCTAGCTAGCTAGCTAGCTAGCTAGCTAGCTAGCTAGCTAGCTAGCTAGC  HHHHHHHHHHHHGGGGGGGGGGGGGGGGGFFFFFCCCCCCCCCCCCCCCC

关键字段： 1. QNAME：读取名称 2. FLAG：比对方向和状态（是否正向链、是否配对、是否未比对等，用位标志编码） 3. RNAME：参考序列名（如 chr1） 4. POS：在参考基因组上的起始位置 5. MAPQ：比对质量（与 Phred 类似的概率编码） 6. CIGAR：比对的紧凑编码（50M 表示 50 个碱基匹配，10S 表示 10 个软剪裁） 7. RNEXT/PNEXT/TLEN：配对末端的信息（对于 RNA-seq 双端测序）

SAM → BAM → CRAM 的进化： - SAM：纯文本，人类可读但巨大（几十 GB） - BAM：SAM 的二进制压缩版，采用 BGZF（block gzip），可随机访问，大小减小 10-20 倍 - CRAM：更激进的压缩，相对参考基因组进行编码，体积再缩 50%，但需要参考基因组文件解压

现代生物信息学几乎都使用 BAM 格式存储。

详情：DNA 高通量测序的完整流程——包括 Sanger 测序、Illumina 测序和 PacBio/Nanopore 长读长测序——详见 第二节·高通量测序。

7.3.4 BED（Browser Extensible Data）：基因组区间的坐标表

如果说 FASTA/FASTQ 记录的是序列本身，BED 就是记录某些生物学意义的区间（genes、peaks、variants 等）的位置。

chr1    1000    2000    gene1   100 +
chr1    3500    4200    gene2   200 -
chr2    5000    6500    peak1   150 .

最小化六列（实际可有 12+ 列）： 1. chrom：染色体名 2. chromStart：起始位置（0-based） 3. chromEnd：结束位置（不含此位点，即半开区间） 4. name：元素名称 5. score：数值评分（如 p-value，或任意 0-1000 的数字） 6. strand：链方向（+ 正向、- 反向、. 未指定）

关键陷阱：BED 采用 0-based 半开区间（[start, end)），而 GTF 采用 1-based 闭区间（[start, end]）。这是数据转换时的常见错误源。

7.3.5 GTF / GFF：基因组注释的百科全书

GTF（Gene Transfer Format）和其前身 GFF（General Feature Format）记录基因组上所有已注释的特征——从基因、转录本、外显子到启动子。

chr1    HAVANA  gene    11869   14409   .   +   .   gene_id "ENSG00000223972"; gene_name "DDX11L1"
chr1    HAVANA  transcript  11869   14409   .   +   .   gene_id "ENSG00000223972"; transcript_id "ENST00000456328"
chr1    HAVANA  exon    11869   12227   .   +   .   gene_id "ENSG00000223972"; transcript_id "ENST00000456328"; exon_number 1
chr1    HAVANA  exon    12613   12721   .   +   .   gene_id "ENSG00000223972"; transcript_id "ENST00000456328"; exon_number 2

9 列结构： 1. seqname：染色体 2. source：数据来源（HAVANA、NCBI 等） 3. feature：特征类型（gene、transcript、exon、CDS 等） 4. start / end：1-based 闭区间 5. score：评分（通常为 .） 6. strand：链方向 7. frame：阅读框（用于 CDS 翻译） 8. attributes：键值对（如 gene_id "XXX"; gene_name "YYY"）

重要提醒：GTF 来自 Ensembl，NCBI 提供的是 GFF3。两者在细节上有差异，混用会导致注释错乱。

7.3.6 数据压缩：体积杀手

在实际应用中，上述所有文本格式都会被压缩： - FASTQ.gz：gzip 压缩，减小 80-90% 体积，但需逐行解压（可线性访问） - BAM：已内置 BGZF 压缩 - GTF.gz：同样使用 gzip

压缩后，原本 50 GB 的 FASTQ 可缩至 5-10 GB，大幅降低存储和网络传输成本。

7.4 分析流程速览：从原始数据到生物学发现

为了帮助理解这些格式如何串联起来，下图展示了一个典型的 RNA-seq 分析流程：

SRA 数据库
  ↓
FASTQ.gz (原始测序数据，数百 GB)
  ↓ [质量控制 + 比对]
BAM 文件 (比对结果，数 GB)
  ↓ [定量表达]
表达量矩阵 (CSV / HDF5，数 MB)
  ↓ [统计分析]
GEO 数据库 (发表结果供全球重用)

每一步都有相应的生物信息学工具（如 fastqc 检查质量、hisat2 比对、featureCounts 定量、DESeq2 统计）。掌握这些格式是读懂工具文档和调试 Pipeline 的前提。

7.5 工具生态前瞻：如何优雅地读取它们

面对这些截然不同的格式，现代生物信息学的法则是：不要重复造轮子。

  • 对于序列文件（FASTA/FASTQ）：在 Python 中，我们使用 Biopython 库的 SeqIO 模块；它能像迭代器一样，每次只吐出一条序列，完美解决内存占用问题。质量检查则使用 fastqc 工具生成 HTML 报告。

  • 对于比对结果（SAM/BAM）：使用 samtools 这个瑞士军刀级工具。它可以在秒级完成 BAM 的排序、索引、统计、转换等操作。在 Python 中，pysam 提供了 Python 封装；在 R 中，Rsamtools 则是标准选择。

  • 对于基因组区间（BED/GTF）：在 R 语言中，Bioconductor 生态下的 GenomicRanges 包是无可争议的王者。它可以将坐标区间转化为代数对象，在一秒钟内完成两套包含几百万个区间的重叠（Overlap）计算。在 Python 中，bedtools 或 pybedtools 提供了类似的功能。

  • 对于表达量矩阵（HDF5 / .h5ad）：在 Python 中使用 AnnData 包（专为单细胞数据设计）；在 R 中使用 Seurat 或 SingleCellExperiment 对象的专门读取接口，它们会自动处理底层的 C++ 库调用，为你呈现出漂亮的数据框对象。

  • 对于公共数据库访问：SRA Toolkit（用于下载 SRA 数据）和 GEOquery（R 包，直接从 GEO 读取表达矩阵）等工具大幅简化了数据获取流程。

7.6 随堂巩固测验

假设你在 NCBI 的 SRA 数据库下载了一个名为 SRR1234567.fastq.gz 的文件，进行了比对后得到 SRR1234567.bam，然后定量为 SRR1234567_counts.csv，并最终上传到 GEO 数据库。

7.6.1 基础概念

问题 1： 这三个文件在生物信息学分析流水线中的上下游顺序是什么？各自存储的是什么信息？

👉 点击查看答案

答案：

三个文件的上下游顺序为：FASTQ → BAM → CSV

文件格式 所处流程阶段 存储的信息类型 数据大小
SRR1234567.fastq.gz 原始数据 原始测序读取（序列 + 质量评分） 数十-数百 GB
SRR1234567.bam 比对结果 读取与参考基因组的比对位置、质量、CIGAR 字符串等 数 GB
SRR1234567_counts.csv 定量结果 每个基因/转录本在该样本中的表达量（通常是 read count 或 FPKM） 数 MB

详细解释：

  1. FASTQ 是从高通量测序仪器直接输出的原始数据。每条读取包含四行信息：序列 ID、实际 DNA/RNA 序列、分隔符和 Phred 质量评分。文件特别大是因为每个碱基都附带质量信息。

  2. BAM 是经过比对软件（如 HISAT2、Bowtie2）处理后的结果。它将每条读取定位到参考基因组的具体坐标位置，包含了比对质量和匹配模式（CIGAR）。BAM 采用二进制压缩，大小相对较小。

  3. CSV 是最终的定量矩阵。通过专门的定量工具（如 featureCounts、RSEM）统计每个基因在每个样本中的读取数量，得到一个行为基因、列为样本的表达矩阵。这是用于后续统计分析和数据库提交的标准格式。

问题 2： 为什么原始的 FASTQ 文件需要被压缩为 .gz 格式？与 BAM 格式的内置压缩相比，有什么区别？

👉 点击查看答案

答案：

FASTQ 压缩的必要性：

  • FASTQ 是纯文本格式，原始大小巨大（50-500 GB）
  • 使用 gzip 压缩可以缩小到 80-90%，即 5-50 GB，显著降低存储和网络传输成本
  • 压缩后仍然可以逐行解压进行流式处理，无需全部解压到内存中

FASTQ.gz vs BAM 的压缩差异：

特性 FASTQ.gz BAM
压缩方式 gzip（通用压缩） BGZF（Block GZipped Format，专门设计）
压缩率 80-90% 10-20 倍（相对 SAM 文本格式）
是否可随机访问 否（流式访问） 是（具有索引 .bai）
是否需要解压使用 需要逐行解压 可直接查询特定坐标
用途 原始数据存储、传输 分析结果存储、快速查询

核心区别：gzip 是通用文本压缩，而 BGZF 是为了兼顾压缩率和随机访问设计的专用格式。

7.6.2 技术细节

问题 3： 在 FASTQ 文件中，质量分数采用 ASCII 编码。如果某个碱基的质量字符是 I，这代表错误概率大约是多少？（参考：ASCII I = 73，Q = ASCII - 33）

👉 点击查看答案

答案：

根据 Phred 质量评分公式计算：

给定信息： - ASCII 字符 ‘I’ 的值为 73 - Phred 公式：Q = ASCII 字符值 - 33 - 错误概率公式：P = 10^(-Q/10)

计算步骤：

  1. 计算 Q 值：

    Q = 73 - 33 = 40

  2. 计算错误概率：

    P = 10^(-40/10) = 10^(-4) = 0.0001 = 0.01%

答案： 字符 ‘I’ 代表的错误概率约为 0.01%（或 1/10000）

补充说明： - Q40 是极高的质量评分 - 通常接受的阈值为 Q20（1% 错误率）或 Q30（0.1% 错误率） - 字符 ‘I’ 对应的质量远超常规要求

问题 4： BED 文件使用 0-based 半开区间，而 GTF 文件使用 1-based 闭区间。假设一个基因在参考基因组上的物理位置是”第 1000 到第 2000 个碱基”，在 BED 和 GTF 中各应该如何记录？

👉 点击查看答案

答案：

坐标系的关键差异：

特性 BED 格式 GTF 格式
坐标系统 0-based 半开区间 [start, end) 1-based 闭区间 [start, end]
含义 start 包含，end 不包含 start 和 end 都包含

具体例子：

假设基因的物理位置是”从第 1000 到第 2000 个碱基”（包括两端）：

BED 记录：

chr1    999 2000    gene1   100 +
  • 起始位置：999（0-based，指向第 1000 个碱基）
  • 结束位置：2000（不含此位点）
  • 跨度：999 到 1999（共 1000 个碱基）

GTF 记录：

chr1    HAVANA  gene    1000    2000    .   +   .   gene_id “gene1”
  • 起始位置：1000（1-based）
  • 结束位置：2000（包含此位点）
  • 跨度：1000 到 2000（共 1001 个碱基）

注意： 两种格式记录的碱基数不同（BED 为 1000，GTF 为 1001），这是数据转换的常见错误源。转换时需要小心调整坐标。

问题 5： 当你使用 samtools view 命令查看 BAM 文件时，你观察到某行的 FLAG 字段值为 16。这个数字代表什么生物学意义？（提示：16 = 0x10 表示反向链）

👉 点击查看答案

答案：

FLAG 值 16 的含义：

FLAG 字段使用位标志（bitwise flag）编码，16 的二进制表示为 00010000：

位置 值 含义
0 0 读取已比对（0）或未比对（1）
1 0 配对读取中的本读取是否配对
2 0 本读取是否未配对
3 0 配对读取的另一端是否未比对
4 1 本读取在反向链（negative strand）
其他 0 （其他信息）

生物学意义：

FLAG = 16 表示： - 该读取来自 DNA 的反向互补链（负链） - 读取已成功比对到参考基因组 - 比对方向为 反向链（reverse strand）

补充说明： - 在双链 DNA 中，一条链是正向（+），另一条是反向（-） - 测序得到的读取可能来自任一条链 - FLAG 值可以是多个位标志的组合，通过加法计算 - 例如：FLAG = 163 (128+32+2+1) 表示配对的第二条读取，在反向链上

7.6.3 应用设计

问题 6： 当你的实验从 10 个老鼠组织样本升级为测量 1,000,000 个独立单细胞的表达量时，为什么你需要将 GSE8888_counts.csv 这样的常规 CSV 格式抛弃，转而寻求 HDF5（如 .h5ad）架构的帮助？说出至少两个技术原因。

👉 点击查看答案

答案：

两个技术原因：

7.6.3.1 原因 1：内存溢出问题

  • CSV 数据结构： 必须将整个文件加载到内存中才能处理
  • 规模增长：
    • 10 个样本：~20,000 基因 × 10 样本 = 20 万个数值，可轻易加载（<100 MB）
    • 100 万细胞：~20,000 基因 × 1,000,000 细胞 = 2×10^10 个数值，需 >100 GB RAM
  • HDF5 优势： 支持分块读取（chunked access），只在需要时加载特定区间的数据

7.6.3.2 原因 2：稀疏矩阵存储

  • 单细胞特性： 大部分基因在大部分细胞中不表达（表达值为 0）
  • CSV 浪费： 存储数百万个 0，造成严重的存储和内存浪费
  • HDF5/稀疏矩阵： 只存储非零值的坐标和值，节省 >95% 空间

具体对比：

CSV 存储 100 万细胞：~150 GB
HDF5（稀疏矩阵）：~5 GB

7.6.3.3 额外优势：

  • 支持元数据存储（细胞类型、质量指标等）
  • 快速子集化：轻松选择特定细胞或基因
  • 版本化存储：.h5ad 格式已成为单细胞分析的行业标准

问题 7： 你计划分析来自不同研究团队的 RNA-seq 数据。其中一个团队提供的 GTF 文件来自 NCBI（GFF3 格式），另一个来自 Ensembl（GTF 格式）。在合并注释时，你可能会遇到什么问题？

👉 点击查看答案

答案：

三大可能问题：

7.6.3.4 问题 1：坐标系统不兼容

  • NCBI GFF3： 1-based 坐标

  • Ensembl GTF： 1-based 坐标

    （虽然都是 1-based，但两者在基因范围定义上有细微差异）

表现： 同一基因在两个注释文件中的坐标可能偏移 1-2 bp，导致比对错误

7.6.3.5 问题 2：基因 ID 命名规范不同

  • NCBI GFF3： 使用 NCBI Gene ID（如 NCBIGene:12345）
  • Ensembl GTF： 使用 Ensembl Gene ID（如 ENSG00000000003）

表现： 同一个生物学基因可能有两个不同的 ID，合并时难以识别

7.6.3.6 问题 3：注释粒度差异

  • NCBI GFF3： 可能缺少某些外显子或启动子等小特征
  • Ensembl GTF： 注释更详细（exon、CDS、start_codon 等分别列出）

表现： 合并后的注释可能不完整或重复

解决方案：

  1. 使用统一的映射数据库（如 biomaRt）将不同 ID 转换为统一标识
  2. 选择一个注释文件作为”主”，用另一个作为补充
  3. 使用专门的注释合并工具（如 bravo 或手写 Python/R 脚本）
  4. 验证合并结果：检查基因数量和外显子统计是否合理

最佳实践： 在分析前期就明确使用哪个注释源，避免后期合并

7.7 关键概念总结：从数据库到实验室

本节涉及的所有要素可以用一个三层模型理解：

┌─────────────────────────────────────────────────────────────┐
│ 第 1 层：全球公共数据库（NCBI、EBI、DDBJ）                │
│  └─ INSDC 24 小时同步机制                                    │
├─────────────────────────────────────────────────────────────┤
│ 第 2 层：文本格式存储（FASTA, FASTQ, SAM, BED, GTF）       │
│  └─ 权衡：人类可读性 vs 存储效率                            │
├─────────────────────────────────────────────────────────────┤
│ 第 3 层：专业工具生态（Biopython, samtools, R Bioconductor）│
│  └─ 承诺：内存高效 + 批量操作 + 统计检验                    │
└─────────────────────────────────────────────────────────────┘

掌握这些格式不是为了”死记硬背”，而是为了： 1. 快速诊断问题：看到一个数据文件，立刻知道它在 Pipeline 中的位置 2. 优雅地集成工具：知道哪些工具吃什么格式、吐什么格式 3. 避免格式陷阱：譬如坐标系的 0-based vs 1-based，链方向的编码等

这就是为什么生物信息学被称为”数据工程与统计学的杂交”——你既要懂算法，也要懂数据的语言。
