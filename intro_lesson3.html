Published

April 10, 2026

5.1 Linux 系统简介

Linux 是一种开源的类 Unix 操作系统内核，由 Linus Torvalds 于 1991 年首次发布。它以其稳定性、安全性和灵活性著称，在服务器、嵌入式系统和桌面环境中广泛应用。在生物信息学领域，Linux 是处理大规模数据分析的首选平台。

5.1.1 Linux 发行版概述

Linux 内核本身并不构成完整的操作系统。为了提供用户友好的体验，开发者基于 Linux 内核创建了各种发行版（Distribution）。这些发行版通常包括： - Linux 内核 - GNU 工具链 - 包管理系统 - 桌面环境（可选） - 预装软件

发行版主要分为两大阵营： - RedHat 系列：企业级，注重稳定性，包管理系统使用 RPM - Debian 系列：社区驱动，注重自由软件，包管理系统使用 DEB

5.1.2 常见 Linux 发行版介绍

5.1.2.1 RedHat 系列

  • Red Hat Enterprise Linux (RHEL)：商业发行版，专为企业环境设计，提供长期支持和商业服务。广泛用于生产服务器。
  • CentOS：RHEL 的社区克隆版，完全兼容 RHEL，但免费。常用于学术和科研环境的高性能计算集群。
  • Fedora：RedHat 的上游发行版，包含最新软件和技术，常用于开发和测试环境。

5.1.2.2 Debian 系列

  • Debian：最古老的 Linux 发行版之一，以稳定性著称。遵循严格的自由软件准则，包管理系统优秀。
  • Ubuntu：基于 Debian 的用户友好发行版，最流行的桌面 Linux 系统之一。Ubuntu Server 版本常用于云计算和生物信息学服务器。
  • Linux Mint：基于 Ubuntu 的发行版，提供更传统的桌面体验。

5.1.2.3 其他常见发行版

  • Arch Linux：滚动更新发行版，注重简洁和最新软件，适合高级用户。
  • Gentoo：源代码编译发行版，提供高度定制化，但安装复杂。

5.1.3 在生物信息学中的应用

不同发行版在生物信息学社区中的应用各有优势：

  • CentOS/RHEL：学术 HPC 集群的首选，许多生物信息学软件在这些系统上经过充分测试。
  • Ubuntu：生物信息学家入门和开发环境的热门选择，易于安装和配置（个人配置首选）。
  • Debian：稳定可靠，适合长期运行的服务器。

在本课程中，我们将重点使用类 CentOS 系统进行演示，但所学知识适用于所有 Linux 发行版。

5.2 告别鼠标，掌握算力的绝对控制权

对于绝大多数人来说，计算机等同于”视窗、图标和鼠标点击”（GUI，图形用户界面）。但在生命科学的大数据时代，一个标准的全基因组测序原始文件（FASTQ）动辄 50GB 甚至 100GB。如果你尝试用 Windows 的记事本或 Excel 去双击打开它，结果只有一个：电脑死机。

在生物信息学和极客的世界里，真正的计算发生在没有屏幕、没有鼠标的服务器机房里。本章通过学习纯文本的命令行界面（CLI, Command Line Interface）直接与计算机的对话。

5.3 为什么生物信息学离不开 Linux？

一般对 Windows 或 macOS 很熟悉，但想要处理高通量测序数据，Linux 操作系统是绕不过去的门槛。

  • 极致的算力释放：Windows 为了维持华丽的图形界面，在后台消耗了大量的 CPU 和内存。而运行在高性能计算集群（HPC）上的 Linux 服务器通常只有黑乎乎的终端，它将 \(99.9\%\) 的硬件资源毫无保留地倾注于数据计算。
  • 无可比拟的稳定性：Linux 系统设计哲学强调”分时、多用户与高并发”，一台配置得当的 Linux 服务器可以连续运行数年而不必重启，完美契合动辄需要计算几十个小时的基因组组装任务。
  • 开源生态与生信工具链：全球 \(99\%\) 以上的权威生物信息学软件（如 BWA, SAMtools, GATK）都是基于 Linux/Unix 环境利用 C/C++ 或 Python 开发的开源软件。许多软件压根就没有 Windows 版本。

5.3.1 Unix 哲学：小工具的巨大力量

Linux 继承自 Unix，其核心设计哲学可以用一句话概括：

“做好一件事，并能与其他程序协作。（Do one thing and do it well.）”

这意味着：grep 只负责搜索，sort 只负责排序，wc 只负责计数——但通过管道（pipe）将它们连接起来，便能完成复杂的数据分析任务，而无需编写一行代码。

5.4 核心架构：Linux 文件系统层级树

在 Windows 中，我们习惯了 C盘、D盘 的概念。在 Linux 中，没有盘符，只有一棵倒立的树。 Linux 的核心哲学之一是：“一切皆文件（Everything is a file）”。

5.4.1 两个最重要的符号：/ 与 ~

  • 根目录（/）：这是整个文件系统树的起点。所有的文件、外接硬盘、光驱，甚至内存，全部都挂载在这个根目录下的不同分支里。
  • 用户家目录（~）：在服务器中，你只是众多用户中的一个。系统会为你分配一个专属的地盘，通常位于 /home/你的用户名/。为了方便，Linux 用波浪号 ~ 来指代你自己的家目录。

5.4.2 关键系统目录说明

了解 Linux 文件系统层级标准（FHS, Filesystem Hierarchy Standard）中的关键目录，有助于定位软件、配置文件和数据：

目录 全称/用途
/bin 基本用户命令（ls、cp、bash 等）
/usr/bin 用户安装的可执行程序
/etc 系统配置文件（软件配置几乎都在这里）
/home 所有普通用户的家目录
/tmp 临时文件，重启后清空
/var 变动数据，如日志文件（/var/log）
/data 或 /scratch HPC 集群上常见的大容量数据目录
/opt 第三方软件手动安装位置

5.4.3 定位逻辑：绝对路径 vs 相对路径

这是初学者最容易犯错的概念： * 绝对路径（Absolute Path）：永远以根目录 / 开头。它就像地理上的经纬度绝对坐标，无论你当前在系统的哪个角落，顺着这个路径一定能找到目标。 * 例如：/home/user/data/sample.fastq * 相对路径（Relative Path）：不以 / 开头。它是基于你当前所在位置的路线指引。 * . 代表当前目录；.. 代表上一级（父）目录。 * 例如：如果你现在在 /home/user/ 下，访问同样的文件的相对路径是 data/sample.fastq。

5.5 基本 bash 操作命令讲解

在掌握了文件系统的基础概念后，让我们学习最常用的 bash 命令。

5.5.1 文件和目录操作

5.5.1.1 ls - 列出目录内容

ls                    # list files in current directory
ls -l                 # long format: permissions, size, timestamp
ls -lh                # human-readable file size (KB, MB, etc.)
ls -a                 # show all files including hidden (dot files)
ls /home/user/        # list contents of a specific path

5.5.1.2 cd - 切换目录

cd /path/to/directory  # change to absolute path
cd relative/path       # change to relative path
cd ..                  # go up one level to parent directory
cd ~                   # go to home directory
cd -                   # go back to previous directory

5.5.1.3 pwd - 显示当前工作目录

pwd  # print working directory, e.g. /home/user/projects

5.5.1.4 mkdir - 创建目录

mkdir new_folder           # create a single directory
mkdir -p path/to/nested    # create nested directories recursively
mkdir data results logs    # create multiple directories at once

5.5.1.5 cp / mv / rm - 复制、移动、删除

cp source.txt dest.txt          # copy a file
cp -r source_dir dest_dir       # copy directory recursively
mv old_name.txt new_name.txt    # rename a file
mv file.txt /path/to/dir/       # move file to another directory
rm file.txt                     # delete a file (irreversible!)
rm -r directory/                # delete directory recursively
rm -i file.txt                  # prompt before deletion (safer)
Warning

rm -rf 是高危命令，不可恢复，执行前务必确认路径正确。在 HPC 上误删数据可能意味着数月实验成果的丢失。

5.5.2 文件查看

5.5.2.1 cat / head / tail - 查看文件内容

cat file.txt             # display entire file
cat -n file.txt          # display with line numbers
head file.txt            # show first 10 lines (default)
head -20 file.txt        # show first 20 lines
tail file.txt            # show last 10 lines
tail -f log.txt          # follow file in real time (useful for logs)

在生物信息学中，FASTQ 文件动辄数 GB，永远不要用 cat 直接显示，应用 head 或 less（分页查看）。

5.5.2.2 less - 分页浏览大文件

less sample.fastq.gz     # page through large file (supports gzip)
# Inside less: use arrows to scroll, / to search, q to quit

5.6 文件权限系统

Linux 是多用户系统，文件权限是保障数据安全的核心机制。

5.6.1 权限的三组与三类

-rwxr-xr--  1  user  group  4096  Jan 1  file.sh

权限字段（rwxr-xr--）分为三组，每组三位：

组别 说明 含义
rwx 文件所有者（Owner） 可读、可写、可执行
r-x 所属组（Group） 可读、不可写、可执行
r-- 其他用户（Others） 只读
  • r（read）= 4，w（write）= 2，x（execute）= 1

5.6.2 chmod 与 chown

chmod 755 script.sh        # set permissions: owner=rwx, group=rx, others=rx
chmod +x script.sh         # add execute permission to all
chmod 600 private_key.pem  # owner read/write only (SSH key requirement)

chown user:group file.txt  # change file owner and group
Note

生物信息学中常见场景：下载的工具脚本需要 chmod +x 才能运行；共享服务器上的数据目录需要正确设置组权限，使实验室成员都能访问。

5.7 管道与重定向：Unix 哲学的精髓

管道（|）和重定向（>、>>）是 Linux 命令行最强大的特性，也是生物信息学数据处理的日常工具。

5.7.1 输出重定向

# Redirect stdout to a file (overwrite)
echo "sample_id,condition" > metadata.csv

# Append stdout to a file
echo "sample1,treated" >> metadata.csv

# Redirect stderr to a file
bwa mem ref.fa reads.fastq 2> bwa_error.log

# Redirect both stdout and stderr
bwa mem ref.fa reads.fastq > alignment.sam 2> error.log

# Discard output entirely
some_verbose_command > /dev/null 2>&1

5.7.2 管道（Pipe）

管道将前一个命令的标准输出直接传给下一个命令的标准输入，无需中间文件：

# Count how many reads are in a FASTQ file (4 lines per read)
cat sample.fastq | wc -l | awk '{print $1/4, "reads"}'

# Find all unique chromosome names in a BED file
cat regions.bed | cut -f1 | sort | uniq

# Check the most common adapter sequences in FastQC report
grep "Adapter" fastqc_data.txt | head -20

# Real pipeline: align and sort in one step (no intermediate SAM file)
bwa mem ref.fa R1.fastq R2.fastq | samtools sort -o sorted.bam
Tip

管道是省磁盘空间的利器。在 NGS 分析中，原始 SAM 文件可能高达数百 GB。通过管道直接从 BWA 输出传入 samtools，可以完全跳过写入磁盘这一步。

5.8 文本处理工具：生信数据操作的瑞士军刀

生物信息学数据（FASTQ、BED、GFF、VCF、TSV）本质上都是纯文本，以下工具是日常必备。

5.8.1 grep - 模式搜索

grep "ATCG" sequences.fa           # find lines containing ATCG
grep -c "^>" sequences.fa          # count FASTA header lines (sequences)
grep -v "^#" variants.vcf          # exclude comment lines (VCF headers)
grep -E "chr[0-9]+\s" regions.bed  # use extended regex

5.8.2 cut - 按列提取

cut -f1,3 gene_counts.tsv          # extract columns 1 and 3 (tab-delimited)
cut -d',' -f2 metadata.csv         # extract column 2 (comma-delimited)
cut -f1 sample.bed | sort | uniq   # get unique chromosomes from BED file

5.8.3 sort 与 uniq - 排序与去重

sort -k1,1 -k2,2n regions.bed      # sort BED file by chr then by position (numeric)
sort -t$'\t' -k5,5rn counts.tsv    # sort by column 5, reverse numeric order

sort gene_list.txt | uniq          # remove duplicate lines
sort gene_list.txt | uniq -c       # count occurrences of each line
sort gene_list.txt | uniq -d       # show only duplicated lines

5.8.4 wc - 统计行/词/字符数

wc -l sample.fastq                 # count lines
echo $(($(wc -l < sample.fastq) / 4))  # count reads in FASTQ
wc -c sample.fastq.gz              # count bytes (file size)

5.8.5 awk - 强大的列操作语言

awk 是处理表格型文本数据最强大的工具，在生物信息学中几乎无处不在：

# Print columns 1 and 4 from a TSV file
awk '{print $1, $4}' gene_counts.tsv

# Filter BED file: keep only regions longer than 1000 bp
awk '$3 - $2 > 1000' regions.bed

# Calculate total coverage from a BED file
awk '{sum += $3 - $2} END {print "Total bp:", sum}' regions.bed

# Add a header and reformat VCF info field
awk 'BEGIN{OFS="\t"} !/^#/ {print $1, $2, $4, $5}' variants.vcf

5.8.6 sed - 流式文本替换

# Replace all occurrences of "chr" with "" (remove chr prefix)
sed 's/^chr//' regions.bed > regions_nochr.bed

# Delete lines starting with # (remove VCF headers)
sed '/^#/d' variants.vcf > variants_noheader.vcf

# In-place replacement (modify file directly)
sed -i 's/sample_A/patient_001/g' metadata.txt

5.9 压缩与解压：与测序数据打交道的必备技能

测序数据几乎总以压缩格式存储，节省约 70% 的磁盘空间。

5.9.1 gzip / gunzip

gzip sample.fastq                  # compress to sample.fastq.gz
gunzip sample.fastq.gz             # decompress
zcat sample.fastq.gz               # view compressed file without decompressing
zcat sample.fastq.gz | head -8     # view first 2 reads without decompressing
gzip -c sample.fastq > sample.fastq.gz  # compress, keep original

5.9.2 tar - 打包与压缩

tar -czvf results.tar.gz results/      # create compressed archive
tar -xzvf results.tar.gz               # extract archive
tar -tzvf results.tar.gz               # list contents without extracting
Note

zcat + 管道是生信中的高频组合：直接在不解压文件的情况下统计、搜索或处理 FASTQ 数据，节省大量磁盘 I/O 时间。

5.10 SSH：远程连接到服务器

在生物信息学实践中，计算任务几乎都在远程 HPC 服务器上运行。SSH（Secure Shell）是连接远程服务器的标准协议。

5.10.1 基本 SSH 连接

ssh username@server.example.edu           # connect to remote server
ssh -p 2222 username@server.example.edu   # connect on non-default port
ssh -i ~/.ssh/id_rsa username@server.edu  # connect with SSH key (passwordless)

5.10.2 SSH 密钥认证（推荐）

密码登录不安全且繁琐，生产环境推荐使用 SSH 密钥对：

# Generate SSH key pair on local machine
ssh-keygen -t ed25519 -C "your_email@example.com"

# Copy public key to remote server (enables passwordless login)
ssh-copy-id username@server.example.edu

# The private key stays on your machine (~/.ssh/id_ed25519)
# The public key is stored on server (~/.ssh/authorized_keys)

5.10.3 文件传输

scp local_file.fastq username@server:/data/     # upload file to server
scp username@server:/results/output.txt ./      # download file from server
scp -r local_dir/ username@server:/data/        # upload directory

# rsync: smarter transfer (only changed files, supports resuming)
rsync -avzP local_dir/ username@server:/data/project/

5.10.4 tmux：保持长时间任务运行

SSH 连接断开后，正在运行的任务会被终止。tmux 允许你的任务在后台持续运行：

tmux new -s rnaseq        # create a new tmux session named "rnaseq"
tmux attach -t rnaseq     # re-attach to an existing session
# Ctrl+b, d               # detach from session (leaves it running)
# Ctrl+b, [               # enter scroll mode (to view output history)

生物信息学软件依赖复杂（Python 版本冲突、库版本不兼容），Conda 通过创建隔离的虚拟环境彻底解决这一问题。

5.11.1 安装 Miniconda / Mamba

# Create a new environment for RNA-seq analysis

# Activate / deactivate environment

5.12 Shell 配置文件：~/.bashrc 与 ~/.bash_profile

每次打开一个新终端，Shell 都会自动读取并执行一些配置文件，这些文件定义了你的工作环境——包括环境变量、命令别名、自定义函数以及 conda 初始化代码。理解这些文件是个性化配置工作环境的关键。

5.12.1 配置文件的加载时机

文件 触发时机 典型用途
~/.bash_profile 登录 Shell（SSH 登录、虚拟控制台） 设置登录时执行一次的配置
~/.bashrc 交互式非登录 Shell（打开新终端窗口） 日常工作配置，别名，函数
~/.profile 登录 Shell（bash 未找到 .bash_profile 时） 跨 Shell 兼容的通用配置
/etc/profile 所有用户登录时 系统级全局配置（需 root 修改）
/etc/bashrc 所有用户交互式 Shell 系统级别名和函数（需 root 修改）
Note

常见做法：在 ~/.bash_profile 末尾加入 source ~/.bashrc，确保登录 Shell 和交互式 Shell 的配置完全一致，避免”SSH 登录后命令找不到”的困惑。

5.12.2 编辑配置文件

# Open ~/.bashrc with nano (beginner-friendly editor)
nano ~/.bashrc

# Or open with vim (more powerful)
vim ~/.bashrc

# After editing, apply changes to current session without reopening terminal
source ~/.bashrc
# Equivalent shorthand
. ~/.bashrc

5.12.3 关键配置内容详解

下面是一个典型的生物信息学工作者的 ~/.bashrc 配置，逐段说明：

# ============================================================
# ~/.bashrc — Bash configuration for interactive shells
# ============================================================

# --- 1. Source system-wide bashrc (keep this if present) ---
if [ -f /etc/bashrc ]; then
    source /etc/bashrc
fi

# --- 2. PATH: tell shell where to find executables ---
# Prepend custom bin directories so they take priority
export PATH="$HOME/bin:$HOME/.local/bin:$PATH"

# Add a manually installed tool (e.g. GATK downloaded to /opt)
export PATH="/opt/gatk-4.4.0:$PATH"

# --- 3. Environment variables ---
export EDITOR="vim"                        # default text editor
export HISTSIZE=10000                      # number of commands in history
export HISTFILESIZE=20000                  # history file size
export HISTCONTROL=ignoredups:erasedups    # no duplicate entries in history

# Bioinformatics data paths (adjust to your server layout)
export REF_DIR="/data/reference/hg38"
export SCRATCH="/scratch/$USER"

# Java heap size for GATK/Picard (set to ~80% of available RAM)
export _JAVA_OPTIONS="-Xmx8g"

# --- 4. Aliases: shortcuts for frequently used commands ---
alias ll='ls -lh --color=auto'
alias la='ls -lah --color=auto'
alias ..='cd ..'
alias ...='cd ../..'

# Safety aliases (prompt before overwriting)
alias cp='cp -i'
alias mv='mv -i'
alias rm='rm -i'

# Bioinformatics shortcuts
alias cdjobs='cd /scratch/$USER/jobs'
alias cddata='cd /data/$USER'
alias sq='squeue -u $USER'             # quick Slurm queue check
alias si='sinfo --format="%P %a %l %D %t %N"'

# --- 5. Custom functions ---

# mkcd: create directory and enter it in one command
mkcd() {
    mkdir -p "$1" && cd "$1"
}

# fqstat: quickly count reads in a FASTQ file
fqstat() {
    echo "File: $1"
    echo "Reads: $(zcat "$1" | wc -l | awk '{print $1/4}')"
    echo "Size:  $(du -sh "$1" | cut -f1)"
}

# bsub_log: submit a slurm job and tail its log immediately
watch_job() {
    local jobid=$(sbatch "$1" | awk '{print $NF}')
    echo "Submitted job: $jobid"
    echo "Waiting for log file..."
    sleep 3
    tail -f slurm-${jobid}.out
}

# --- 6. Conda initialization (auto-added by conda init) ---
# >>> conda initialize >>>
# !! Contents within this block are managed by 'conda init' !!
__conda_setup="$('/opt/miniconda3/bin/conda' 'shell.bash' 'hook' 2> /dev/null)"
if [ $? -eq 0 ]; then
    eval "$__conda_setup"
else
    if [ -f "/opt/miniconda3/etc/profile.d/conda.sh" ]; then
        . "/opt/miniconda3/etc/profile.d/conda.sh"
    else
        export PATH="/opt/miniconda3/bin:$PATH"
    fi
fi
unset __conda_setup
# <<< conda initialize <<<

# --- 7. Module system (common on HPC clusters) ---
# Load frequently used system modules at login
module load java/17
# module load cuda/11.8   # uncomment if using GPU jobs

# --- 8. Custom prompt (PS1): show username, host, current dir, git branch ---
parse_git_branch() {
    git branch 2>/dev/null | grep '\*' | sed 's/* //'
}
export PS1='\[\033[01;32m\]\u@\h\[\033[00m\]:\[\033[01;34m\]\w\[\033[33m\]$( b=$(parse_git_branch); [ -n "$b" ] && echo " ($b)")\[\033[00m\]\$ '

5.12.4 逐段重点说明

PATH 变量是最重要的环境变量，Shell 根据它来按顺序搜索可执行文件。当你安装了两个版本的 samtools，系统运行的是哪个？取决于哪个目录在 PATH 中更靠前：

# Check which samtools is actually being used
which samtools
# /opt/miniconda3/envs/bio_env/bin/samtools

# See the full PATH
echo $PATH | tr ':' '\n'    # print each directory on its own line

别名（alias）的陷阱：别名 alias rm='rm -i' 会让每次删除都要确认，但在脚本中调用 rm 时别名不生效（脚本中的 Shell 是非交互式的），因此脚本里的 rm 仍是危险的——在脚本中始终显式加 -i 或先做路径检查。

自定义函数 vs 别名：别名只能替换简单命令，不能接受参数逻辑；函数则是完整的 Shell 脚本片段，可以接收参数 $1、$2，使用 if/for 等结构。

5.12.5 修改后立即生效

# Apply changes in current terminal session
source ~/.bashrc

# Verify a new alias works
alias sq
# alias sq='squeue -u user123'

# Verify PATH was updated
echo $PATH | tr ':' '\n' | head -5

# Check a new environment variable
echo $REF_DIR
# /data/reference/hg38
Tip调试技巧

如果 source ~/.bashrc 后报错，可逐段注释排查：

# Temporarily disable a section by commenting it out
# export _JAVA_OPTIONS="-Xmx8g"

# Or run bash in verbose mode to see each line being executed
bash --verbose ~/.bashrc 2>&1 | head -30

5.12.6 配置文件的版本控制

和分析代码一样，配置文件也应纳入版本控制（俗称 dotfiles 管理）：

# Create a dotfiles repository
mkdir ~/dotfiles && cd ~/dotfiles
cp ~/.bashrc ./bashrc
cp ~/.vimrc ./vimrc    # if you use vim
git init
git add .
git commit -m "Initial dotfiles"
git remote add origin https://github.com/yourname/dotfiles.git
git push -u origin main

# On a new server, restore your config in one step
git clone https://github.com/yourname/dotfiles.git ~/dotfiles
ln -sf ~/dotfiles/bashrc ~/.bashrc     # symlink instead of copy
source ~/.bashrc

这样，无论在哪台服务器上登录，你都能在几分钟内恢复熟悉的工作环境。

5.13 服务器调度系统简介

在生物信息学领域，许多计算任务（如基因组组装、RNA-seq 分析、变异检测）需要大量的计算资源。这些任务往往运行在高性能计算集群（HPC, High Performance Computing）上，而不是单台个人电脑。为了高效管理和分配这些资源，HPC 系统通常配备了作业调度系统（Job Scheduler）。

5.13.1 为什么需要作业调度系统？

想象一下，一个生物信息学实验室有数百名研究人员同时提交计算任务。调度系统解决了这些问题：

  • 队列管理：任务按顺序排队等待执行；
  • 资源分配：根据任务需求分配 CPU 核心、内存、GPU 等；
  • 优先级控制：重要任务可以优先执行；
  • 负载均衡：避免单点过载；
  • 监控和报告：实时查看任务状态和系统负载。

5.13.2 常见的作业调度系统

在学术和科研环境中，最常用的调度系统包括：

  1. Slurm：最流行，开源，广泛用于学术 HPC 集群；
  2. PBS/Torque：历史悠久，许多老系统仍在使用；
  3. LSF (Load Sharing Facility)：商业系统，功能强大；
  4. SGE (Sun Grid Engine)：开源，但逐渐被 Slurm 取代。

本课程重点介绍 Slurm。

5.13.3 Slurm 基本命令

5.13.3.1 sbatch - 提交批处理作业

作业脚本示例（rnaseq_align.sh）：

#!/bin/bash
#SBATCH --job-name=rnaseq_analysis
#SBATCH --output=rnaseq_%j.out    # %j is replaced by job ID
#SBATCH --error=rnaseq_%j.err
#SBATCH --time=48:00:00           # max wall time
#SBATCH --mem=32GB                # total memory
#SBATCH --cpus-per-task=8         # CPU cores
#SBATCH --partition=compute

# Load software modules
module load star/2.7.10
module load samtools/1.17

# Activate conda environment
conda activate rnaseq

# Set working directory
cd /data/project/rnaseq/

# Run alignment
echo "Starting STAR alignment..."
STAR \
  --runThreadN 8 \
  --genomeDir /data/ref/hg38/star_index \
  --readFilesIn sample_R1.fastq.gz sample_R2.fastq.gz \
  --readFilesCommand zcat \
  --outSAMtype BAM SortedByCoordinate \
  --outFileNamePrefix results/sample1_

echo "Alignment complete!"
sbatch rnaseq_align.sh            # submit job script

5.13.3.2 常用状态查看命令

squeue -u $USER                   # view your jobs in queue
squeue -p compute                 # view all jobs in a partition
sinfo                             # view cluster node and partition status
scancel 12345                     # cancel job ID 12345
scancel -u $USER                  # cancel all your jobs
sacct -j 12345 --format=JobID,State,Elapsed,MaxRSS  # job usage stats after completion

5.13.3.3 常见状态码

状态 含义
PD (Pending) 排队等待中
R (Running) 正在运行
CG (Completing) 即将完成
CD (Completed) 已完成
F (Failed) 运行失败（查看 .err 文件）
OOM 内存溢出（需增加 --mem）

5.13.4 作业数组：批量提交多样本任务

对于需要对多个样本重复相同分析的场景，作业数组（Job Array）可以一次提交多个任务：

#!/bin/bash
#SBATCH --job-name=multi_sample
#SBATCH --array=1-24              # submit 24 jobs (one per sample)
#SBATCH --cpus-per-task=4
#SBATCH --mem=16GB
#SBATCH --time=12:00:00

# Read sample name from a list file
SAMPLE=$(sed -n "${SLURM_ARRAY_TASK_ID}p" sample_list.txt)

echo "Processing sample: ${SAMPLE}"
fastp -i raw/${SAMPLE}_R1.fastq.gz -I raw/${SAMPLE}_R2.fastq.gz \
      -o clean/${SAMPLE}_R1.fastq.gz -O clean/${SAMPLE}_R2.fastq.gz \
      --thread 4 --detect_adapter_for_pe
sbatch array_job.sh               # submits 24 independent jobs automatically

5.13.5 最佳实践

  • 合理估算资源：不要请求过多资源（浪费积分/排队时间），也不要请求过少（OOM 失败）；
  • 先测试小任务：正式提交前用 1-2 个样本验证脚本正确性；
  • 使用 tmux + 实时日志：tail -f rnaseq_12345.out 监控运行进度；
  • 清理临时文件：及时删除中间文件，HPC 磁盘配额有限；
  • 记录软件版本：使用 conda env export 或 module list 记录分析环境，保证可重复性。
