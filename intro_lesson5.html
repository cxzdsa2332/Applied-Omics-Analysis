7  探索生物大数据的基石：数据库网络与核心文件格式

Published

April 27, 2026

7.1 引言：数据的汪洋与打水的桶

在开始敲代码之前，我们必须回答两个问题：“全世界的生物数据存放在哪里？” 以及 “我们把数据下载到本地后，它们是以什么面貌存在的？” 生物信息学是一个高度强调“开源与共享”的学科。任何一篇顶级期刊论文发表的前提，都是必须将原始测序数据上传到全球公共数据库中。今天，我们将先认识这些全球数据枢纽，再深入拆解承载这些海量信息的核心文件格式。

7.2 宏观世界：全球生物数据基础设施

在计算科学中，我们可以把这些数据库理解为生物学界的云存储集群，它们共同构成了 INSDC（国际核酸序列数据库合作组织，International Nucleotide Sequence Database Collaboration）。INSDC 的三个协作伙伴（NCBI、EBI、DDBJ）采用24 小时同步机制：任何一个数据库收到的新数据，必须在 24 小时内同步到其他两个伙伴。这确保了全球研究者无论向哪个数据库提交，都能被全世界同步感知。

{{Wiki:International_Nucleotide_Sequence_Database_Collaboration}}

7.2.1 NCBI（美国国家生物技术信息中心）

这是生物信息学领域的绝对霸主，相当于生物学界的 Google。它旗下有几个我们需要极其熟悉的子数据库：

  • SRA（Sequence Read Archive）：原始高通量测序数据的公共存档库，存储全球所有 FASTQ/BAM 格式的原始读段，数据量已超过数十 PB。每篇高通量测序论文发表时均须在此提交原始数据。
  • GEO（Gene Expression Omnibus）：处理后的表达量矩阵仓库。研究者将算好的”基因表达量矩阵”（CSV 或 HDF5 文件）上传至此，是进行元分析（Meta-analysis）和数据挖掘的首要资源库。
  • GenBank / RefSeq：分别对应原始提交序列（GenBank）和经专家审核的高质量参考序列（RefSeq）。RefSeq 中的 accession 以 NM_（mRNA）、NP_（蛋白质）、NC_（染色体）等前缀区分，是变异注释时引用坐标的标准来源。
  • dbSNP / ClinVar：存储人类基因组中的已知变异（SNP、Indel）及其临床意义判定，是临床遗传学和群体基因组学不可或缺的注释资源。

如何从 NCBI 程序化下载数据：

NCBI 提供 Entrez E-utilities API，允许通过 HTTP 请求直接检索和下载数据，无需进入网页界面。例如，通过 esearch + efetch 组合可以批量下载某物种的全部参考序列。SRA 数据则通过 sra-tools 中的 prefetch 和 fasterq-dump 命令高效获取。

7.2.2 EBI（欧洲生物信息学研究所）与 Ensembl

EBI 是欧洲的对应机构，旗下管理多个专业数据库：ENA（European Nucleotide Archive） 是 SRA 的欧洲镜像，ArrayExpress 是 GEO 的对应资源，UniProt 是全球最权威的蛋白质序列与功能数据库。

Ensembl 是 EBI 与 Sanger Institute 联合维护的基因组浏览器和注释项目。如果说 NCBI 偏向”收纳一切”，Ensembl 则在基因组注释方面做到了极致精准——其注释不仅包含基因坐标，还包括转录本异构体、调控元件、直系同源关系及跨物种比较基因组信息。下载参考基因组 GTF 注释文件时，Ensembl 是首选来源。

7.2.3 UCSC Genome Browser

UCSC（加州大学圣克鲁斯分校）基因组浏览器是基因组可视化领域的另一重要平台。它的核心价值在于：将基因组注释、保守性得分、染色质开放性、转录因子结合位点等数十种”轨道（Track）”叠加在同一基因组坐标上进行可视化比对，是探索某一基因组区域全貌的直观工具。UCSC 使用自己的坐标体系（0-based），其常用格式 BED 和 bigWig 均源于此。

7.2.4 数据库选择策略

需求 推荐数据库
下载原始 FASTQ 数据 ENA（国内网络最快）或 SRA
获取已处理表达矩阵 GEO 或 ArrayExpress
下载参考基因组序列 NCBI RefSeq 或 Ensembl
下载基因注释 GTF Ensembl（优先）或 NCBI GFF3
查询蛋白质功能 UniProt
变异临床意义注释 ClinVar / dbSNP
基因组区域可视化探索 UCSC Genome Browser 或 Ensembl Browser

7.3 文本的哲学：序列与坐标格式（ASCII）

从数据库下载数据后，我们最先接触到的是纯文本（Plain Text）格式。它们牺牲了存储效率，但换取了极高的“人类可读性”。

7.3.1 FASTA（.fasta / .fa）：参考基因组的字典

最古老、最基础的序列存储格式，本质是一个键值对（Key-Value Pair）集合。 * Key：以 > 开头，包含序列 ID（如染色体编号）和可选的描述信息。 * Value：紧随其后的纯碱基（A/T/C/G）或氨基酸序列。

>chr1 Homo sapiens chromosome 1, GRCh38
NNNNNNNNNNNNNNNTAACCCTAACCCTAACCCTAACCCTAACCCTAACCCTA...
>chr2 Homo sapiens chromosome 2, GRCh38
NNNNNNNNNNNNNNNTAACCCTAACCCTAACCCTAACCCTAACCCTAACCCTA...

关键特性： - 极其简洁，易于解析 - 每行通常限制在 50-80 字符以提高可读性 - 广泛用于参考基因组、蛋白序列库（如 UniProt）

7.3.2 FASTQ（.fastq / .fq）：原始测序数据的真实面貌

当高通量测序仪器（Illumina、PacBio 等）输出数据时，每条读取片段（read）都包含序列本身和质量评分。FASTQ 格式正是为此而生。

@SRR1234567.1 reads from sample A
AGCTTAGCTAGCTACGTACGTACGTACGTACGTACGTACGTACGTACGTACGTAC
+
IIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIII
@SRR1234567.2 reads from sample A
GCTAGCTAGCTAGCTAGCTAGCTAGCTAGCTAGCTAGCTAGCTAGCTAGCTAGC
+
HHHHHHHHHHHHHHHHGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGGFFFFFCCCC

四行结构： 1. 行 1：@ 前缀 + 读取 ID + 可选描述 2. 行 2：实际序列 3. 行 3：+ 分隔符（有时后跟 ID，有时为空） 4. 行 4：质量分数（与序列等长）。采用 Phred 质量评分系统：ASCII 字符编码表示概率

质量分数的秘密： - Phred 质量 Q = -10 × log₁₀(P)，其中 P 是碱基错误的概率 - 字符 I (ASCII 73) ≈ Q30，表示 0.1% 错误率（极高质量） - 字符 # (ASCII 35) ≈ Q2，表示 63% 错误率（极低质量） - 一般接纳的阈值：Q20（1% 错误率）或 Q30

这就是为什么原始数据文件的大小达到 数十到数百 GB——每个碱基都附带质量信息。

延伸阅读：Phred 质量分数系统的详细原理和 Illumina 测序流程，已在 第二节·高通量测序原理 中详细介绍。

7.3.3 SAM / BAM / CRAM：序列比对的金字塔

从 FASTQ 文件出发，下一步是将这些短读取与参考基因组进行比对（alignment）。这个过程输出 SAM（Sequence Alignment Map）格式。

SAM 格式（可读文本版）：

@HD VN:1.0  SO:coordinate
@SQ SN:chr1 LN:248956422
read1   0   chr1    1000    60  50M *   0   0   AGCTTAGCTAGCTACGTACGTACGTACGTACGTACGTACGTACGTACG    IIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIII
read2   16  chr1    2000    30  40M10S  *   0   0   GCTAGCTAGCTAGCTAGCTAGCTAGCTAGCTAGCTAGCTAGCTAGCTAGC  HHHHHHHHHHHHGGGGGGGGGGGGGGGGGFFFFFCCCCCCCCCCCCCCCC

关键字段： 1. QNAME：读取名称 2. FLAG：比对方向和状态（是否正向链、是否配对、是否未比对等，用位标志编码） 3. RNAME：参考序列名（如 chr1） 4. POS：在参考基因组上的起始位置 5. MAPQ：比对质量（与 Phred 类似的概率编码） 6. CIGAR：比对的紧凑编码（50M 表示 50 个碱基匹配，10S 表示 10 个软剪裁） 7. RNEXT/PNEXT/TLEN：配对末端的信息（对于 RNA-seq 双端测序）

SAM → BAM → CRAM 的进化： - SAM：纯文本，人类可读但巨大（几十 GB） - BAM：SAM 的二进制压缩版，采用 BGZF（block gzip），可随机访问，大小减小 10-20 倍 - CRAM：更激进的压缩，相对参考基因组进行编码，体积再缩 50%，但需要参考基因组文件解压

现代生物信息学几乎都使用 BAM 格式存储。

详情：DNA 高通量测序的完整流程——包括 Sanger 测序、Illumina 测序和 PacBio/Nanopore 长读长测序——详见 第二节·高通量测序。

7.3.4 BED（Browser Extensible Data）：基因组区间的坐标表

如果说 FASTA/FASTQ 记录的是序列本身，BED 就是记录某些生物学意义的区间（genes、peaks、variants 等）的位置。

chr1    1000    2000    gene1   100 +
chr1    3500    4200    gene2   200 -
chr2    5000    6500    peak1   150 .

最小化六列（实际可有 12+ 列）： 1. chrom：染色体名 2. chromStart：起始位置（0-based） 3. chromEnd：结束位置（不含此位点，即半开区间） 4. name：元素名称 5. score：数值评分（如 p-value，或任意 0-1000 的数字） 6. strand：链方向（+ 正向、- 反向、. 未指定）

关键陷阱：BED 采用 0-based 半开区间（[start, end)），而 GTF 采用 1-based 闭区间（[start, end]）。这是数据转换时的常见错误源。

7.3.5 GTF / GFF：基因组注释的百科全书

GTF（Gene Transfer Format）和其前身 GFF（General Feature Format）记录基因组上所有已注释的特征——从基因、转录本、外显子到启动子。

chr1    HAVANA  gene    11869   14409   .   +   .   gene_id "ENSG00000223972"; gene_name "DDX11L1"
chr1    HAVANA  transcript  11869   14409   .   +   .   gene_id "ENSG00000223972"; transcript_id "ENST00000456328"
chr1    HAVANA  exon    11869   12227   .   +   .   gene_id "ENSG00000223972"; transcript_id "ENST00000456328"; exon_number 1
chr1    HAVANA  exon    12613   12721   .   +   .   gene_id "ENSG00000223972"; transcript_id "ENST00000456328"; exon_number 2

9 列结构： 1. seqname：染色体 2. source：数据来源（HAVANA、NCBI 等） 3. feature：特征类型（gene、transcript、exon、CDS 等） 4. start / end：1-based 闭区间 5. score：评分（通常为 .） 6. strand：链方向 7. frame：阅读框（用于 CDS 翻译） 8. attributes：键值对（如 gene_id "XXX"; gene_name "YYY"）

重要提醒：GTF 来自 Ensembl，NCBI 提供的是 GFF3。两者均使用 1-based 坐标，但属性字段的键名、层级关系的表达方式存在差异，混用会导致注释解析错乱。建议在项目开始时即确定使用单一注释来源，并将版本号（如 Ensembl 109）记录在分析记录中。

{{Wiki:General_feature_format}}

7.3.6 VCF（Variant Call Format）：基因组变异的通用语言

VCF 是记录基因组变异（Variant）的标准格式，包括单核苷酸多态性（SNP）、小片段插入缺失（Indel）及结构变异（SV）。它由千人基因组计划（1000 Genomes Project）推动制定，如今已成为所有变异分析流程的终点输出格式。

{{Wiki:Variant_Call_Format}}

##fileformat=VCFv4.2
##reference=GRCh38
##FILTER=<ID=PASS,Description="All filters passed">
##FORMAT=<ID=GT,Number=1,Type=String,Description="Genotype">
##FORMAT=<ID=DP,Number=1,Type=Integer,Description="Read Depth">
#CHROM  POS     ID        REF  ALT  QUAL  FILTER  INFO           FORMAT    SAMPLE1
chr1    925952  rs1234567 G    A    50    PASS    DP=35;AF=0.5   GT:DP     0/1:35
chr1    981931  .         AT   A    30    PASS    DP=22;AF=1.0   GT:DP     1/1:22
chrX    155701  .         C    CTAG  99   PASS    DP=40;AF=0.5   GT:DP     0/1:40

文件结构：

VCF 由三个区块构成：

  1. 元信息行（Meta-information）：以 ## 开头，定义文件版本、参考基因组、INFO/FORMAT/FILTER 字段的含义。这些定义是自描述的关键——没有它们，后续字段的值将无法解读。

  2. 列标题行：以单个 # 开头，固定为 CHROM POS ID REF ALT QUAL FILTER INFO FORMAT SAMPLE1 [SAMPLE2 ...]，多样本时每个样本各占一列。

  3. 数据行：每行一个变异位点，字段含义如下：

列名 含义 示例
CHROM 染色体（1-based 坐标） chr1
POS 变异位置（1-based） 925952
ID 已知变异 ID（dbSNP 等） rs1234567 或 .
REF 参考基因组碱基 G
ALT 替代等位基因 A（SNP）或 CTAG（Indel）
QUAL 变异质量得分（Phred-scaled） 50
FILTER 过滤状态 PASS 或过滤原因
INFO 额外注释信息（键=值对） DP=35;AF=0.5
FORMAT 样本字段的格式定义 GT:DP
SAMPLE 各样本的基因型信息 0/1:35

基因型（Genotype，GT）字段解读：GT 字段是 VCF 的核心信息，编码方式如下：

GT 值 含义
0/0 纯合参考型（Homozygous reference）
0/1 杂合突变（Heterozygous）
1/1 纯合突变（Homozygous alternate）
0/2 参考型与第二个替代等位基因的杂合
./. 基因型无法判断（缺失数据）

/ 分隔的两个数字分别代表二倍体生物的两条同源染色体上的等位基因，0 代表参考等位基因，1、2 等正整数代表 ALT 列中对应位置的替代等位基因。

BCF（Binary Call Format） 是 VCF 的二进制压缩版（类似 BAM 之于 SAM），体积约缩小 5–10 倍，由 bcftools 工具链处理，支持随机访问和快速过滤。

VCF 1-based 坐标与 Indel 位点表示

VCF 采用 1-based 坐标，与 GTF 一致，但与 BED 不同。记录 Indel 时，VCF 有一个特殊规则：锚点碱基（Anchor Base）。删除 AT→A（即删除 T）时，VCF 以 REF=AT、ALT=A 表示，其中第一个 A 是锚点——它本身不变，但它与后续碱基共同定义了缺失的范围。这一规则保证了 Indel 的唯一表示，但在坐标转换时需格外注意。

7.3.7 WIG / bedGraph / bigWig：覆盖度与信号轨道

上述格式均记录离散的基因组特征（序列、变异、区间）。而许多实验（ChIP-seq 染色质免疫共沉淀、ATAC-seq 染色质开放性、RNA-seq 覆盖度）的分析结果是连续的数值信号，需要专用格式存储。

{{Wiki:ChIP-sequencing}}

bedGraph：BED 格式的扩展，第四列记录该区间的数值信号（如覆盖深度、富集得分）。

chr1    0       100     0.0
chr1    100     200     12.5
chr1    200     350     45.8
chr1    350     400     3.2

Wiggle（WIG）：另一种文本格式，支持 fixedStep（等间距）和 variableStep（可变间距）两种写法，常用于保守性得分（PhyloP、GERP）等全基因组信号的存储。

bigWig：bedGraph/WIG 的二进制索引版本，是现代基因组浏览器（UCSC、IGV）显示信号轨道的标准格式。其核心优势在于支持随机区间查询——无需读入全文件，即可快速提取特定染色体区域的数值。

# Convert bedGraph to bigWig (requires chromosome size file)
bedGraphToBigWig coverage.bedgraph chrom.sizes output.bw

# Query a specific region from bigWig
bigWigSummary output.bw chr1 10000 20000 10
格式 类型 随机访问 文件大小 典型用途
bedGraph 文本 否 大 中间过渡格式
WIG 文本 否 大 保守性得分
bigWig 二进制 是 小 可视化轨道、信号统计

7.3.8 HDF5 / AnnData（.h5ad）：单细胞时代的矩阵格式

传统的 CSV/TSV 表格格式在样本量达到数万至数百万（单细胞测序）时面临根本性挑战。HDF5（Hierarchical Data Format 5） 是专为大规模科学数据设计的二进制格式，支持分块存储（chunking）、压缩、元数据嵌入和快速随机访问。

{{Wiki:Hierarchical_Data_Format}}

AnnData（.h5ad） 是 Python anndata 包定义的单细胞数据容器，基于 HDF5 构建，已成为 Python 单细胞分析生态（Scanpy）的标准数据结构：

AnnData 对象的内部结构：
  .X          → 核心表达量矩阵（细胞 × 基因），稀疏矩阵格式
  .obs         → 细胞级元数据（细胞类型、样本来源、质控指标等）
  .var         → 基因级元数据（基因名、染色体、是否高变基因等）
  .obsm        → 降维结果（PCA、UMAP 坐标等）
  .uns         → 非结构化信息（颜色方案、版本记录等）

.h5ad 格式能够在单个文件中同时保存数百万细胞的表达矩阵、降维坐标、聚类标签和质控记录，且文件体积远小于等价的 CSV（稀疏矩阵存储仅记录非零值）。

在 R 语言生态中，SingleCellExperiment 和 Seurat 对象扮演类似角色，底层分别使用 HDF5 或专有 .rds 文件存储。

7.3.9 Tabix：文本格式的随机访问索引

基因组分析中，研究者经常需要快速查询”某染色体某区间内有哪些变异/注释”。对压缩文本文件逐行扫描极为低效，Tabix 提供了一种通用解决方案：对 bgzip 压缩的 BED、GFF、GTF、VCF 等格式构建坐标索引（.tbi 文件），实现 \(O(\log n)\) 级别的区间查询。

# Step 1: Sort and bgzip compress
sort -k1,1 -k2,2n variants.vcf | bgzip > variants.vcf.gz

# Step 2: Build Tabix index (generates variants.vcf.gz.tbi)
tabix -p vcf variants.vcf.gz

# Step 3: Query a specific region in milliseconds
tabix variants.vcf.gz chr1:1000000-2000000

Tabix 索引机制已成为 VCF、BED、GTF 等格式在生产环境中的标准配置，尤其在分布式存储（云端 S3 对象存储）中，可以仅下载目标区间对应的数据块，无需传输整个文件。

7.3.10 数据压缩：体积杀手

在实际应用中，上述所有文本格式都会被压缩： - FASTQ.gz：gzip 压缩，减小 80-90% 体积，但需逐行解压（可线性访问） - BAM：已内置 BGZF 压缩 - GTF.gz：同样使用 gzip

压缩后，原本 50 GB 的 FASTQ 可缩至 5-10 GB，大幅降低存储和网络传输成本。

7.4 分析流程速览：格式如何在流程中流动

格式不是孤立存在的——每种格式都是流程中某个工具的输出，同时也是下一个工具的输入。理解这条”格式链”，是读懂工具文档、调试 Pipeline 的根本前提。

7.4.1 RNA-seq 差异表达分析流程

SRA / ENA 数据库
  ↓  [sra-tools prefetch + fasterq-dump]
FASTQ.gz（原始测序数据，每样本数十 GB）
  ↓  [FastQC 质量评估 → fastp/Trimmomatic 清洗]
FASTQ.gz（清洗后）
  ↓  [HISAT2 / STAR 比对，使用 GTF 注释引导拼接]
SAM → BAM（排序 + 索引，每样本数 GB）
  ↓  [featureCounts / HTSeq 基于 GTF 坐标定量]
counts.csv（基因 × 样本表达量矩阵，数 MB）
  ↓  [DESeq2 / edgeR 差异表达统计]
results.csv + volcano plot
  ↓  [上传至 GEO，论文发表]

7.4.2 基因组变异检测流程（GATK 最佳实践）

FASTQ.gz
  ↓  [BWA-MEM 比对]
SAM → BAM（排序）
  ↓  [Picard MarkDuplicates 去重]
BAM（去重后）
  ↓  [GATK HaplotypeCaller]
GVCF（中间格式，每样本）
  ↓  [GATK GenotypeGVCFs 联合基因型分型]
VCF（原始变异）
  ↓  [VQSR / hard filtering 过滤]
VCF（高质量变异）
  ↓  [VEP / ANNOVAR 功能注释，使用 GFF3/GTF + dbSNP]
annotated VCF → 临床报告 / 群体分析

7.4.3 单细胞 RNA-seq 流程

FASTQ.gz（含细胞条形码 + UMI）
  ↓  [Cell Ranger / STARsolo / Alevin 比对 + 去重]
原始计数矩阵（h5 格式，细胞 × 基因）
  ↓  [Scanpy / Seurat 质控过滤]
AnnData .h5ad（过滤后，每样本数万细胞）
  ↓  [归一化 → 高变基因选择 → PCA → UMAP → Leiden 聚类]
.h5ad（含降维坐标和聚类标签）
  ↓  [标记基因识别 → 细胞类型注释]
最终注释的 .h5ad → 发表 / 存入 CELLxGENE 数据库

黄金法则：格式版本必须全局统一

一个项目中使用的参考基因组（如 GRCh38 p14）、GTF 注释（如 Ensembl 109）、dbSNP（如 b156）三者必须来自同一版本体系。混用版本会导致坐标偏移、基因 ID 不匹配等隐性错误，且往往在分析末期才显现，代价极高。建议在项目的第一个脚本中用注释记录所有参考文件的来源、版本和 MD5 校验值。

7.5 工具生态前瞻：如何优雅地读取它们

面对这些截然不同的格式，现代生物信息学的法则是：不要重复造轮子。

  • 对于序列文件（FASTA/FASTQ）：在 Python 中，我们使用 Biopython 库的 SeqIO 模块；它能像迭代器一样，每次只吐出一条序列，完美解决内存占用问题。质量检查则使用 fastqc 工具生成 HTML 报告。

  • 对于比对结果（SAM/BAM）：使用 samtools 这个瑞士军刀级工具。它可以在秒级完成 BAM 的排序、索引、统计、转换等操作。在 Python 中，pysam 提供了 Python 封装；在 R 中，Rsamtools 则是标准选择。

  • 对于基因组区间（BED/GTF）：在 R 语言中，Bioconductor 生态下的 GenomicRanges 包是无可争议的王者。它可以将坐标区间转化为代数对象，在一秒钟内完成两套包含几百万个区间的重叠（Overlap）计算。在 Python 中，bedtools 或 pybedtools 提供了类似的功能。

  • 对于表达量矩阵（HDF5 / .h5ad）：在 Python 中使用 AnnData 包（专为单细胞数据设计）；在 R 中使用 Seurat 或 SingleCellExperiment 对象的专门读取接口，它们会自动处理底层的 C++ 库调用，为你呈现出漂亮的数据框对象。

  • 对于公共数据库访问：SRA Toolkit（用于下载 SRA 数据）和 GEOquery（R 包，直接从 GEO 读取表达矩阵）等工具大幅简化了数据获取流程。

7.6 课后自测 (Post-Lesson Quiz)

  1. 以下三个文件 SRR1234567.fastq.gz、SRR1234567.bam、SRR1234567_counts.csv 在典型 RNA-seq 分析流程中的先后顺序是什么？各自存储的是什么层次的信息？

  2. 在 FASTQ 文件中，某碱基的质量字符为 I（ASCII 73）。请计算其 Phred Q 值，以及对应的碱基错误率。

  3. BED 文件使用 0-based 半开区间，GTF 文件使用 1-based 闭区间。若一个基因物理上占据参考基因组第 1000 至第 2000 个碱基（含两端），在 BED 和 GTF 中起止坐标各应如何填写？两种格式记录的碱基数是否相同？

  4. VCF 文件中，某行的 GT 字段值为 0/1，REF 为 G，ALT 为 A。请解释该样本在这个位点的基因型，并说明数字 0 和 1 分别代表什么。

  5. gzip（.gz）与 BGZF（BAM 内置压缩）均是基于 gzip 算法的压缩格式，但二者在实际使用中有一个关键差异。请描述该差异，并解释为什么 BAM 格式需要这种特性。

  6. 你正在分析一个单细胞 RNA-seq 数据集，含 500,000 个细胞、25,000 个基因。如果将表达矩阵存为 CSV 格式（每个数值 4 字节），理论上需要多少存储空间？而实际单细胞数据中大约 95% 的值为零，改用 HDF5 稀疏矩阵存储后，空间约缩减至多少？

  7. 你需要从一个已建立 Tabix 索引的 VCF 文件中快速提取 chr17:7,500,000-7,700,000 范围内的所有变异（该 VCF 文件大小为 20 GB）。与逐行扫描全文件相比，Tabix 查询在原理上为何能显著加速这一操作？

参考答案 (Reference Answers)

  1. 顺序为 FASTQ → BAM → CSV。FASTQ 存储原始测序读段（序列 + 每碱基质量评分），是仪器的直接输出；BAM 存储每条读段与参考基因组的比对结果（坐标、方向、CIGAR 字符串），由比对软件生成；CSV 存储每个基因在每个样本中的表达计数，由定量工具汇总比对结果得到，是统计分析的输入。

  2. Q = 73 − 33 = 40。错误率 P = 10^(−40/10) = 10^(−4) = 0.01%。Q40 属于极高质量，远高于常用阈值 Q30（0.1% 错误率）。

  3. BED 记录：chr1 999 2000（start = 999，0-based；end = 2000，半开区间，不含第 2001 位）；该区间跨度 = 2000 − 999 = 1000 bp。GTF 记录：chr1 ... 1000 2000（start = 1000，1-based；end = 2000，闭区间，含第 2000 位）；跨度 = 2000 − 1000 + 1 = 1001 bp。两者不同：BED 记录 1000 个碱基，GTF 记录 1001 个碱基，这是坐标系差异导致的，数据转换时须调整。

  4. GT = 0/1 表示该样本在此位点为杂合子（Heterozygous）。数字 0 代表参考等位基因（REF = G），数字 1 代表第一个替代等位基因（ALT = A）。即该个体在两条同源染色体上一条携带 G，另一条携带 A。

  5. gzip 将整个文件压缩为连续的压缩流，必须从头解压才能到达任意位置（顺序访问）。BGZF 将文件切分为若干独立的小压缩块（默认约 64 KB），每个块可独立解压；配合 .bai 索引文件，可以直接定位并解压特定染色体区域所对应的块，实现随机访问。BAM 格式需要随机访问是因为基因组分析常需快速查询特定区间的比对读段，例如可视化某基因区域或统计局部覆盖度，顺序扫描整个 BAM 文件效率极低。

  6. 理论全矩阵大小 = 500,000 × 25,000 × 4 字节 = 5×10^10 字节 ≈ 50 GB。若 95% 值为零，非零元素比例约 5%，稀疏存储仅需记录非零值的行列坐标和数值（COO 或 CSR 格式），存储量约缩减至 50 GB × 5% ≈ 2.5 GB，加上 HDF5 本身的元数据和压缩，实际文件通常在 1–5 GB 范围内。

  7. Tabix 为 bgzip 压缩文件建立了坐标索引（.tbi 文件），记录每个基因组区间对应的文件字节偏移量。查询时，Tabix 先在索引中二分查找目标区间（O(log n)），定位到对应的 bgzip 数据块偏移，然后仅读取和解压那几个数据块（可能仅数 KB）。逐行扫描需要读取全部 20 GB；而 Tabix 查询仅需读取极少量字节，差距在数万倍以上。
